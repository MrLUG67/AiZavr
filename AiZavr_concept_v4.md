# AiZavr — Концепция проекта

**Версия:** v4
**Назначение документа:** Описание идеи AiZavr — для онбординга новых участников, объяснения концепции, маркетинговых материалов. Тактические и технические детали разработки (открытые вопросы, технологический стек, roadmap, журнал решений) — в файле `AiZavr_spec_v1.json`.
**Изменения v4:** открытая архитектура с Plugin API, блокнот «Личность», open-source MIT/Apache, уточнение потоков карточек.

---

## 1. НАЗНАЧЕНИЕ

### Что это
AiZavr — **рабочее пространство для длительных творческих и исследовательских диалогов с LLM**. Не «ещё один чат-клиент», а среда, решающая структурные проблемы глубокого общения с ИИ: линейность диалога, переполнение контекста, отсутствие оркестрации специализированных моделей.

### Для кого
Power-users: исследователи, разработчики, писатели, аналитики, дизайнеры, консультанты. Те, кто ведёт долгие сессии с ИИ и упирается в ограничения стандартных чатов.

### Позиционирование
**«IDE для творческих диалогов с ИИ»** — по аналогии с VS Code как IDE для кода, а не «текстовый редактор». Чат — лишь один компонент рабочего процесса.

### Происхождение названия
AiZavr — от «динозавры». Метафора эволюции диалога: динозавры были сильной и развитой ветвью жизни, но многие её линии оказались тупиковыми. Так и в диалогах с ИИ есть направления, от которых стоит уйти раньше, чем дойдёшь до тупика. Ветвление позволяет не терять наработанное и пробовать альтернативы.

## 2. УСТРОЙСТВО И ЭЛЕМЕНТЫ ИНТЕРФЕЙСА

### Основная парадигма
Диалог хранится не как линейный список, а как **дерево узлов**. Каждое сообщение (пользователя или модели) — узел, у которого может быть несколько детей-ветвей. Активная ветка — это путь от корня до текущего листа.

### Главные элементы

**Поле основного диалога.** Видна только активная ветка, выпрямленная как обычный чат. Свёрнутые блоки — компактными плашками. Артефакты (картинки, видео, аудио) — с превью прямо в потоке.

**Навигатор по дереву.** Визуализация структуры с подсветкой активной ветки. Клик по узлу — переход к нему.

**Индикатор контекста («светофор»).** Рядом с полем ввода. Цветом и цифрой показывает заполнение контекстного окна. Пороги: зелёный <40%, жёлтый 40–70%, оранжевый 70–90%, красный >90%. При наведении — разбивка по блокам.

**Действия над сообщением.** На каждом узле: создать ветку, свернуть от/до с уровнем, закрепить (защитить от автосжатия), перейти как к активной точке.

**Четыре кнопки сжатия:** L0 Hidden / L1 Brief / L2 Summary / L3 Tag.

**Кнопка «найти тангенсы».** Запускает анализ диалога отдельной (дешёвой) LLM — она подсвечивает блоки-кандидаты на сжатие. Пользователь решает сам.

**Панель подтверждения генерации артефактов.** Когда основная модель предлагает сгенерировать что-то (картинку, видео, аудио), появляется панель с промптом, параметрами и стоимостью. Кнопки: «Сгенерировать» / «Подправить» / «Не сейчас».

### Принципы UI
- Никаких всплывающих окон без явного действия.
- Никаких звуковых уведомлений.
- Никакой пульсации, мерцания.
- Никаких принудительных действий — даже на 99% заполненности пользователь решает сам.
- Минимум элементов на основном экране — всё детальное в настройках.

## 3. ОСОБЕННОСТИ И ФИЧИ

### 3.1. Ветвление диалогов

Любой узел может стать точкой ответвления. Создав ветку от Q3, пользователь получает Q3.1, и дальше диалог идёт независимо. Все ветки сохраняются, переключение мгновенно.

Главная техническая основа: **LLM-модели stateless**. Они не хранят историю — её хранит приложение. При каждом запросе AiZavr отправляет выпрямленную линейную ветку от корня. Модель «не знает» про дерево. Это делает решение универсальным для любой модели.

Применения:
- «Что если переформулировать?» — пробуем варианты без потери исходного.
- Сравнение моделей: одну точку диалога отправляем в разные модели, ветки рядом.
- Защита от тупиков — всегда можно вернуться к развилке.

### 3.2. Сжатие «лирических отступлений»

В долгом диалоге неизбежны побочные обсуждения, которые съедают токены и отвлекают модель. AiZavr позволяет сворачивать их с регулируемой потерей информации — по аналогии с JPEG-компрессией.

**Четыре уровня:**

| Уровень | Что | Экономия | Сохранено |
|---------|-----|----------|-----------|
| L0 Hidden | Только UI-свёртка | 0% | Всё |
| L1 Brief | Пересказ 2–4 абзаца | ~85% | Структура и итоги |
| L2 Summary | Один абзац | ~95% | Только суть |
| L3 Tag | Однострочная метка | ~99% | Только факт обсуждения |

**Ключевая архитектурная находка:** сжатая версия — это **ветка от исходного узла**. Все уровни сжатия живут как сёстры-ветки, переключение мгновенно. Возврат к оригиналу = переключение на родительскую ветку.

```
[Q23–A34] исходный диапазон
├── Полный текст
├── L1 Brief
├── L2 Summary
└── L3 Tag
```

**Защищённые блоки** (не сворачиваются ни на каком уровне): код, числа/версии/идентификаторы, финальные решения, закреплённые сообщения.

### 3.3. Индикатор контекста

Два индикатора, которые часто путают:

**Объём** (всегда работает, бесплатно). Простой счётчик токенов на стороне приложения. Пороги в процентах от окна модели (адаптируется автоматически: Claude 200k и Llama 8k имеют разные пороги при одной шкале).

**Замусоренность** (по запросу, отдельная кнопка). LLM-вызов с промптом «найди тангенсы». Использует дешёвую модель. Подсвечивает кандидатов в UI.

Накопительный счётчик трат — видимый, в реальном времени, с разбивкой по типам операций (диалог / уплотнение / анализ / артефакты).

### 3.4. Оркестрация специализированных моделей

AiZavr — диспетчер для разных типов ИИ-контента. Долгий диалог с умной моделью (Sonnet, GPT) для проработки идеи → передача промпта в специализированный генератор (Recraft, Flux, Runway, ElevenLabs) → результат в дереве как новый узел.

**Поддерживаются типы:** изображения, видео, музыка и звук, озвучка, распознавание речи, поиск в вебе, исполнение кода, 3D-модели, документы.

**UX-паттерн:** модель предлагает, пользователь подтверждает. Не автоматизм (риск трат) и не ручные кнопки на каждом сообщении (ломает поток). Гибрид: модель формирует промпт и просит подтверждение.

**Артефакт = узел дерева.** Вся механика веток и сжатия применима. Не понравилась картинка? Ветка-сестра, обе сохранены. Сравнение генераторов? Параллельные ветки.

**Стоимость артефактов** — на порядки выше токенов. Поэтому: подтверждение обязательно, стоимость показывается до клика, опциональные лимиты с предупреждением, отдельная строка в счётчике.

### 3.5. Killer-сценарий: «креативный директор + исполнители»

Связка всех фич образует уникальный рабочий процесс:
- Обсуждаем замысел с Sonnet (ветка A).
- Не нравится концепция — возвращаемся к развилке, идём иначе (ветка B).
- Для каждой концепции генерируем по 3 варианта картинки → 6 дочерних узлов.
- Сравниваем рядом, выбираем лучший.
- Дорабатываем выбранный. Отвергнутые сжимаем до L3.

Это не «чат с картинками». Это рабочий процесс, которого нет ни у одного существующего продукта.

### 3.6. Организация бесед: блокноты и карточки знаний

Корни бесед организуются в **блокноты** — контейнеры с возможностью вложенности (блокнот внутри блокнота). Это организация *над* беседами, не путать с деревом узлов *внутри* беседы.

```
Блокнот "Досуг" (карточка знаний)
├── Блокнот "Кино" (своя карточка)
├── Блокнот "Театр" (своя карточка)
│   ├── беседа "что посмотреть в пятницу"
│   └── беседа "впечатления от Вишнёвого сада"
└── беседа "планы на выходные" (прямо в корне)
```

**Карточка знаний.** У каждого блокнота есть структурированный живой документ с ключевыми фактами по его теме. По умолчанию пустой. Это простое решение задачи «обмена знаниями между беседами» — без сложного RAG, под полным контролем пользователя.

**Заливка.** Отдельной кнопкой можно «залить» в карточку ключевую информацию из беседы. Это не копирование, а извлечение смысла: пользователь пишет «"Вишнёвый сад" понравился» → модель предлагает факт «Чехов — любимый драматург» → пользователь подтверждает или правит. Разовую информацию («что показывают в кино сегодня») заливать не нужно — она уйдёт в беседу и не будет засорять карточку.

**Структура и слияние.** Карточка разбита на разделы. Новые факты *сливаются* (merge) с существующими, а не дописываются в конец, — иначе через полгода карточка превратится в простыню с дублями.

**Наследование сверху вниз.** В голову беседы подмешивается цепочка карточек по пути от корневого блокнота до блокнота беседы. Беседа про театр получает карточку «Досуг» + карточку «Театр». Беседа про кино — «Досуг» + «Кино», но не «Театр». Карточки не накапливаются снизу вверх — это предотвращает перегрузку токенами.

**Карточка — свойство блокнота, не беседы.** Подтягивается динамически из текущего блокнота-родителя. При перетаскивании беседы в другой блокнот со следующего открытия подмешивается карточка нового блокнота — автоматически, без вопросов.

**Метка миграции.** При перемещении беседы между блокнотами в её историю вставляется служебная метка «[контекст изменён: ААА → БББ]». Прошлые ответы модели остаются в дереве — метка позволяет модели понять происхождение возможных несоответствий, а пользователю — отладить странные ответы. Миграция не разрушает историю беседы.

## 4. НАСТРОЙКИ

### Философия
**Разумные дефолты + полная возможность переопределить.** Новичок получает работающую систему за 30 секунд. Опытный — настраивает каждый аспект. Дефолты прозрачны: всегда видно, что и почему выбрано, и где это поменять.

Принципиально: «программа не решает за пользователя, что ему хорошо».

### Прогрессивное раскрытие
- При первом запуске — мастер из 3 экранов (способ подключения, ключ, модель).
- Основные настройки — 3–4 пункта на первой странице.
- Продвинутые — на отдельной вкладке.

### Три слоя конфигурации

**Слой 1: Провайдеры.** Список подключённых источников моделей. Поддерживаются:
- OpenRouter — рекомендуется для тех, кто не хочет разбираться (один ключ → 400+ моделей).
- Прямые ключи: Anthropic, OpenAI, Google и т.д.
- Локальные модели через Ollama, LM Studio.
- Произвольный OpenAI-совместимый endpoint.

**Слой 2: Модели.** Агрегированный список из всех провайдеров. Если одна модель доступна через несколько — показаны все варианты.

**Слой 3: Роли.** Назначение моделей под задачи: основной диалог, уплотнение L1/L2/L3, анализ замусоренности, генерация заголовков, артефакты по типам и т.д. У каждой роли — модель по умолчанию + возможность оверрайда.

### Пресеты моделей уплотнения
Не плоский список, а **3 пресета + «свой»**:
- Экономный (Haiku, Gemini Flash) — ~$0.001 за операцию.
- Сбалансированный (Sonnet, GPT-средний) — ~$0.01.
- Качественный (Opus, GPT-топ) — ~$0.05.

### Профили
Именованные пресеты настроек, переключаются одной кнопкой: «Code», «Research», «Casual» и т.д.

### Безопасность ключей
- В системном keychain ОС (Windows Credential Manager / macOS Keychain / Linux Secret Service).
- Не в plain text.
- Никакой телеметрии с ключами, никаких облачных бэкапов по умолчанию.

### Экспорт / импорт настроек
JSON-файл, с ключами или без. Возможность делиться удачными конфигами.

## 5. ДРУГИЕ ХАРАКТЕРИСТИКИ

### Платформа
MVP — Windows Desktop. В перспективе — кроссплатформенность (macOS, Linux) и опциональная облачная синхронизация. Подробности технологического выбора в `spec.json`.

### Open-source
AiZavr — открытый проект. Лицензия MIT или Apache 2.0. Код на GitHub. Это не вынужденное решение — это философское: продукт создаётся для пользы, а не ради монетизации. Открытость оставляет дверь для open-core модели в будущем (платная синхронизация и т.п.) без изменения лицензии.

### Открытая архитектура с Plugin API
AiZavr строится как **расширяемая платформа**, а не монолит. Принцип: ядро маленькое и стабильное, всё остальное — плагины. Ближайшие аналоги: VS Code, Emacs, Obsidian.

Что может быть плагином:
- Провайдеры моделей (новый LLM-сервис = новый плагин).
- Стратегии уплотнения (другой промпт для L2, другой алгоритм сжатия).
- Обработчики карточек (своя логика формирования и обобщения).
- Интеграции с внешними API (новый генератор изображений = новый плагин).
- UI-компоненты и темы оформления.

Формальный публичный Plugin API — после MVP. До того — принцип «plugin-friendly»: ключевые функции с первого дня пишутся как изолированные модули с чёткими интерфейсами, которые позже легко открыть.

Это отвечает на фундаментальную проблему: система слишком большая, чтобы один человек реализовал всё. Открытый Plugin API позволяет сообществу достроить то, что нужно конкретным пользователям.

### Блокнот «Личность» — корневой контекст
Самый верхний блокнот системы, в который вложены все остальные. Его карточка знаний содержит глобальные характеристики пользователя: стиль мышления, ключевые предпочтения, важные жизненные контексты. В силу механизма наследования эта карточка автоматически подмешивается в голову **всех** бесед системы — любая беседа, где бы она ни лежала, знает базовый контекст «кто я».

Пользователь сам решает, что записывать в «Личность». Система не переписывает её автоматически.

### Два потока карточек знаний
Карточки работают в двух независимых направлениях:

**Формирование (снизу вверх):** карточка блокнота наполняется из его бесед и вложенных блокнотов. При подъёме факт обобщается — конкретное «любит Чехова» на уровне «Театр» становится «тяготеет к классике» на уровне «Досуг» и «ценит глубину» на уровне «Личность».

**Использование (сверху вниз):** при открытии беседы в её голову подмешивается вся цепочка карточек от корня до блокнота беседы. Беседа про театр получает контекст «Личность» + «Досуг» + «Театр» одновременно.

Индикатор свежести (светофор) показывает, насколько карточка устарела с момента последнего пересчёта. Пересчёт — по кнопке, решение за пользователем.

### Хранение данных
По умолчанию полностью локально. Опциональная синхронизация — отключаемая, с E2E-шифрованием. Возможна работа полностью офлайн с локальными моделями через Ollama.

### Безопасность и приватность
- Диалоги хранятся локально.
- API-ключи в системном keychain.
- Никакой обязательной телеметрии.
- Никакой подмены моделей — пользователь видит, кто отвечает, и платит провайдеру напрямую своим ключом.

### Бизнес-модель
На старте — **не подписка с автопродлением**. Пользователь платит провайдерам LLM напрямую. AiZavr не посредник в платежах. Потенциальная монетизация в будущем — только через добровольный платный сервис (синхронизация и т.п.), ядро остаётся бесплатным и открытым.

### Конкурентное поле и отличие
В обсуждении упомянуты: KnowTree, Loom, Forky, Tangents.chat, tldraw branching-chat, Microsoft Agent Framework, LangGraph, ChatGPT/Claude (с их скрытым ветвлением), askaichat.app и подобные.

**Уникальное предложение AiZavr:**
1. Древовидное хранение + видимый граф в самостоятельном клиенте.
2. Многоуровневое сжатие с защищёнными блоками — нет в потребительских продуктах.
3. Оркестрация специализированных моделей с обсуждением замысла — нет прямых конкурентов.
4. Блокноты с карточками знаний и корневым контекстом «Личность».
5. Открытая архитектура с Plugin API — расширяемая платформа, а не монолит.
6. Полная мульти-провайдерность с собственными ключами.
7. Прозрачность и уважение к пользователю — антитеза скам-сервисам в нише.

Подробнее по конкурентам, рискам и метрикам успеха — в `spec.json`.

## КЛЮЧЕВЫЕ АРХИТЕКТУРНЫЕ ИДЕИ

Семь «озарений» обсуждения, которые стоит держать в голове:

1. **LLM stateless.** Приложение хранит дерево, модель ничего не знает о ветках. Это даёт универсальность.
2. **Сжатие = ветвление.** Одна и та же механика покрывает оба сценария. Архитектурная экономия.
3. **Артефакт = узел дерева.** Оркестрация моделей естественно ложится на существующую структуру.
4. **Два потока карточек.** Формирование (снизу вверх, обобщение) и использование (сверху вниз, наследование) — разные потоки, разные направления, не конфликтуют.
5. **Прогрессивное раскрытие.** Настройки разворачиваются под пользователя, а не обрушиваются сразу.
6. **Расширяемая платформа.** Ядро маленькое, всё остальное — плагины. AiZavr как «Emacs для LLM-диалогов».
7. **Open-source как стратегия.** Не вынужденное решение, а способ привлечь сообщество, которое достроит то, что один человек не успеет.
