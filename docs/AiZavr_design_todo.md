# AiZavr — Замечания по интерфейсу (design TODO)

**Назначение:** копилка замечаний и идей по UI/UX, отложенных до этапа отладки дизайна. Структура проекта в приоритете — дизайн причёсывается позже, разом, когда устоятся UI-паттерны. Этот файл — чтобы не терять замечания, возникающие по ходу разработки.

**Статус:** черновик, пополняется по мере работы.

## Архитектурный принцип: единое поле ввода

Ключевая идея, связывающая пункты 2 и 4 ниже. Вместо нескольких полей ввода (обычный промпт + всплывающее поле ветвления + потенциальное поле для новой карточки) — **одно поле ввода внизу экрана**. Меняется только «точка привязки» результата в зависимости от контекста:

- обычный промпт → привязка к листу активной ветки;
- ветка по кнопке «+» → привязка к выбранному A-узлу;
- новая карточка в режиме развилки → привязка к тому же A-узлу-развилке.

Поле одно, логика привязки переключается контекстом. Это упрощает и UI (нет дублирующих полей), и код. Стоит держать как принцип при рефакторинге ввода.

## Список замечаний

### 1. Экспорт беседы (функционал, не только дизайн)
Возможность выделить часть беседы и кнопкой выгрузить её файлом. Форматы: PDF, DOCX и др.
- Пересекается с Q-008 в spec (экспорт ветки/дерева наружу).
- Отложено: это полноценная фича, не правка дизайна. Зафиксировать здесь как UI-вход в неё (выделение + кнопка), реализация — по роадмапу (v0.4, экспорт в Markdown как первый шаг).

### 2. Кнопка «+» использует основное поле ввода
Сейчас при нажатии «+» под A-узлом выскакивает отдельное поле ввода. Заменить на:
- при нажатии «+» под ответом An низ этого ответа автоматически подскролливается так, чтобы оказаться прямо над строкой ввода;
- далее используется то же основное поле ввода, что и для обычных запросов;
- второе (всплывающее) поле убрать.

### 3. Карточки вариантов — внизу экрана
Ряд карточек альтернативных запросов выводить внизу, под окончанием ответа LLM, а не вверху (как сейчас, между индикаторами глубины и сообщениями).

### 4. Добавление новой ветки из режима выбора карточек
Если на ответ Ak уже есть, например, три варианта запросов (три карточки), нужна простая возможность задать 4-й вариант прямо из меню выбора карточек — создать ещё одну ветвь, не выходя из режима развилки.
- Использует тот же принцип единого поля ввода (см. выше): ввод нового запроса в контексте развилки привязывается к A-узлу-развилке.

### 5. Раскладка режима развилки
Когда режим выбора карточек НЕ вызван:
- сверху — ответ LLM;
- ниже — кнопки дерева и «+»;
- под ними — строка ввода.

Когда режим выбора карточек вызван:
- текст ответа LLM уходит вверх на высоту меню карточек;
- ряд альтернативных ветвей располагается между ответом и строкой ввода.

То есть карточки вклиниваются между ответом и полем ввода, а не перекрывают контент и не висят сверху.

### 6. Мягкое удаление ветви (архитектура, не дизайн)
Удаление ветви делается «фиктивным»: узлу ставится признак `is_deleted`, ветвь скрывается из выдачи, физически ничего не удаляется (в духе D-004 — ничего не теряется, потом можно сделать «корзину»/восстановление). Вход в UI — пункт «Удалить» в контекстном меню карточки (символ ⋮); каркас меню уже готов, пункт добавляется после реализации.

Что тянет за собой (нужно проработать перед реализацией):
- поле `is_deleted` в `nodes` → миграция 004;
- фильтрация удалённых в `get_children`, `get_branch`, в подсчёте для иконки развилки (`children_count` — считать ли удалённых);
- что делать с `active_child`, если удалена активная ветвь — переключить на живую сестру; какую;
- случай удаления единственной оставшейся ветви.

Кандидат в Q-0xx в spec при следующем обновлении — это открытый архитектурный вопрос, а не правка дизайна.
### 7. Retry прямо в заглушке (unanswered_placeholder)
Когда ответ от LLM не пришёл (краш, обрыв связи, уход в другую ветку), под
холостым Q остаётся серая плашка «ответ не получен» (узел node_type =
unanswered_placeholder). Сейчас плашка — некликабельный индикатор: чтобы
получить ответ, юзер шлёт вопрос заново, что вешает новый Q под заглушку и
плодит дубль вопроса в ленте.

Нужно: кнопка «Повторить запрос» прямо на серой плашке. Она НЕ создаёт новый Q,
а заново дёргает LLM для существующего холостого Q и перезаписывает ЕГО заглушку
(resolve_answer по тому же placeholder_id). Мусора нет, длинный вопрос
копипастить не надо.

Что тянет за собой:
- новая команда ядра cmd_retry_answer(placeholder_id) — собирает LLM-цепочку до
  холостого Q (правило среза служебных пар не применяется к самому целевому Q),
  зовёт LLM, при успехе resolve_answer перезаписывает заглушку;
- UI: кнопка на плашке, состояние loading на время запроса;
- решить, что показывать, если retry тоже провалился (плашка остаётся, можно
  жать снова).

Не делаем сейчас: сначала стабилизируем переход на send_user_message /
resolve_answer, retry — отдельным ходом поверх.

### 8. Иконки маркеров (визуальный якорь)
Кроме имени "#N" и комментария, дать маркеру маленькую иконку по выбору
пользователя из набора (~128-256 штук: фрукты, символы, знаки — !, ?, звёздочка
и т.п.). Мотивация: узнавание работает быстрее чтения. Найти "яблочко" в списке
маркеров мгновенно, "#3" — надо прочитать и сопоставить. Особенно полезно для
будущей навигации по списку маркеров (визуальный якорь > текстовый).

Реализация (когда дойдём, не сейчас):
- НЕ хранить 128 файлов-картинок. Использовать icon-font / emoji: в БД лежит
  строковый id иконки (marker.icon = "apple"), рендерится из набора.
- В стеке уже есть lucide-react — можно брать оттуда, либо emoji-набор.
- Тянет: колонка icon в таблицу markers (миграция), пикер иконок в UI при
  постановке/редактировании маркера.
- Уникальность не нужна — иконки могут повторяться (это якорь, не идентификатор).

Не делаем сейчас: сначала базовый маркер (#N + комментарий) работает, иконки —
поверх.

### 9. Погашенный маркер (серый след)
Идея: маркер отработал (точка больше не нужна как активная), но комментарий и
номер #N остаются видимы в ленте серыми/disabled — как след «тут был маркер».
Третье состояние маркера наряду с есть/нет.

Почему отложено: это НОВОЕ хранимое состояние (не выводится из ссылок, как
D-067 — пользователь решает погасить вручную). Тянет миграцию 006 (поле вроде
dimmed_at в markers), правки create/delete_marker, отдельный UI. Проектировать
на свежую голову, не в конец сессии.

Открытый вопрос к проработке: чем «погашенный» отличается от «свежепоставленного,
ещё не использованного» — оба ни на что не ссылаются. Нужно определить, что
формально значит «отработал».

### 10. message-actions: выравнивание кнопок (СДЕЛАНО, сессия 7)
При раскрытой высокой textarea комментария маркера кнопки ⑂/+/⚑ растягивались
по вертикали (флекс align-items: stretch по умолчанию). Исправлено:
`.message-actions { align-items: flex-start }`. Кнопки прижаты к верху,
не тянутся за textarea. Зафиксировано как выполненное, не как TODO.

### 11. Рефактор: mod.rs → плоские имена модулей (отложено)
Сейчас у каждого Rust-модуля файл называется `mod.rs` (`src-tauri/src/db/mod.rs`,
`markers/mod.rs`, `tree/mod.rs`, `llm/mod.rs`, `keychain/mod.rs`) — десяток
неразличимых вкладок `mod.rs` в редакторе.
Цель: перейти на идиоматический стиль Rust 2018 — `X/mod.rs` → `X.rs`
(т.е. `db/mod.rs` → `db.rs`, `markers/mod.rs` → `markers.rs` и т.д.), имя файла
= имя модуля. Подмодули, если появятся, кладутся в одноимённую папку рядом
(`db.rs` + `db/schema.rs`).
ВАЖНО: НЕ `db_mod.rs` (подчёркивание-суффикс — Rust такой файл как модуль не
видит, сборка сломается). Только `db.rs`.
Делать через `git mv src-tauri/src/db/mod.rs src-tauri/src/db.rs` (сохранить
историю/blame), удалить опустевшие папки, ОТДЕЛЬНЫМ коммитом
"refactor: mod.rs → плоские имена модулей" — чисто механика, ноль изменений логики.
Проверить: после переименования `cargo build` зелёный; `mod db;` и т.п. в lib.rs
менять НЕ нужно (имя модуля то же, меняется только расположение файла).
