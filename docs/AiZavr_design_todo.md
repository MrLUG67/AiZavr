## Предпосылки оркестрации артефактов

К оркестрации (панель подтверждения, роли, вызов генераторов) **не переходим**, пока не готова инфраструктура отображения и открытия результатов. Spec: **D-091**, **D-092**.

### A. Плашка медиафайла в ленте беседы (D-092) — В РАБОТЕ / MVP
Узел `node_type='artifact'` — **отдельная плашка** в `DialogView`, не пузырь Q/A.

**Обязательно на плашке:**
- **Иконка смысла** — `media_kind`: картинка / звук / видео / 3D-модель / документ / прочее (хост-native, напр. lucide-react);
- **Явное расширение** — `.png`, `.wav`, `.glb` и т.д.: читаемая метка формата, не приглушённая подпись мелким шрифтом.

**Опционально:** имя файла, размер в KB/MB.

**Данные:** `nodes.extra.artifact` — как минимум `media_kind`, `filename`, `extension` (или `mime` → extension), `size_bytes`.

**Действие:** кнопка / двойной клик «Открыть» → D-091 (материализация + ОС).

**Где править:** `DialogView.tsx` + `App.css`; типы в `dialog/types.ts` / `NodeView`; при загрузке ветки — прокинуть `extra` для artifact-узлов.

**Сделано (сессия):** `ArtifactPlaque`, `MediaKindIcon`, ветка в `DialogView`, стили, кнопка 📎 в шапке, `cmd_attach_artifact` / `cmd_open_artifact`, модуль `artifacts/`.

**Осталось:** тесты, оркестрация (генерация), tier-2 превью.

### B. Материализация и открытие через ОС (D-091) — частично в коде
`cmd_open_artifact` + копирование в `app_data/artifacts/` при вставке. Осталось: отдельная `cmd_materialize_artifact`, крупные файлы по пути в `extra`.

## Опорный механизм оркестрации: function calling (tool use)

**Ключевая находка (2026-06-26).** Текстовая модель сама картинку не рисует, но через **function calling** (Gemini: `tools[].function_declarations`; OpenAI/OpenRouter: `tools` / `tool_calls`) она выступает **диспетчером**: распознаёт намерение и возвращает структурированный вызов инструмента вместо текста. Исполняет вызов **наша программа**, подставляя результат обратно. Это и есть оркестровка из концепции (зоопарк моделей, D-014): текстовая модель = «мозг/маршрутизатор», image-модель = «руки».

**Пример (Gemini).** Запрос несёт декларацию инструмента:
```jsonc
"tools": [{ "function_declarations": [{
  "name": "generate_image",
  "description": "Вызывай, если пользователь просит создать/нарисовать/сгенерировать изображение.",
  "parameters": { "type": "OBJECT", "properties": {
    "prompt": { "type": "STRING", "description": "Подробное описание картинки (англ.) для генератора." },
    "aspect_ratio": { "type": "STRING", "enum": ["1:1","16:9","4:3"] }
  }, "required": ["prompt"] }
}]}]
```
Модель отвечает НЕ текстом, а вызовом:
```jsonc
"parts": [{ "functionCall": { "name": "generate_image",
  "args": { "prompt": "A cute ginger cat in a spacesuit...", "aspect_ratio": "1:1" } } }]
```

**Полный цикл (многошаговый обмен в рамках одного запроса пользователя):**
1. Шлём `contents` + `tools` (декларация `generate_image`).
2. Модель возвращает `functionCall` (вместо текста).
3. **Наш код** вызывает image-плагин с `args.prompt` → получает байты (наш готовый путь `extra.attachments` → плашки).
4. Шлём 2-й запрос: исходный диалог + `functionCall` модели + `functionResponse` («картинка готова»).
5. Модель пишет финальный текст; картинка уже у нас — прикрепляем.

**Почему важно:**
- закрывает п. 13: даже текстовая модель (Flash-Lite) «выдаёт картинки» косвенно — через вызов инструмента;
- кросс-провайдерно (Gemini `functionCall` ≈ OpenAI `tool_calls`) — архитектура единая;
- предпосылка готова: плашка + открытие артефактов (D-091/D-092) уже умеют показывать результат tool-вызова.

**Чего стоит (объём, не мелкая правка):**
- петля tool-calling: отправка `tools` → разбор `functionCall` → исполнение → `functionResponse` → повторный вызов; лимит рекурсии;
- реестр инструментов приложения и привязка к плагинам/ролям (роль «генератор изображения»);
- 2+ обращения к моделям на одну картинку (стоимость/задержка), обработка ошибок инструмента;
- провенанс: чей `functionCall` (модель-диспетчер) и чем исполнен (image-плагин/модель) — двухслойно, как у сжатия (D-065).

**Связь:** это механизм, на котором строится оркестровка (раздел «Предпосылки оркестрации артефактов»); предпосылки A/B (D-091/D-092) теперь закрыты — инфраструктурно оркестровка разблокирована. Кандидат на отдельное решение в spec (D-xxx) при старте работ.

## Архитектурный принцип: единое поле ввода

Ключевая идея, связывающая пункты 5–6 ниже. Вместо нескольких полей ввода (обычный промпт + всплывающее поле ветвления + потенциальное поле для новой карточки) — **одно поле ввода внизу экрана**. Меняется только «точка привязки» результата в зависимости от контекста:

- обычный промпт → привязка к листу активной ветки;
- ветка по кнопке «+» → привязка к выбранному A-узлу;
- новая карточка в режиме развилки → привязка к тому же A-узлу-развилке.

Поле одно, логика привязки переключается контекстом. Это упрощает и UI (нет дублирующих полей), и код. Стоит держать как принцип при рефакторинге ввода.

**Частично реализовано:** «+» и альтернативный запрос уже используют общий composer (`branchingFromId`). В режиме развилки composer сейчас скрывается — см. п. 5–6.

## Активный список

### 1. Плагин «Экспорт» — отчёт по линейному диапазону маркеров (PDF / DOC)
**Не** экспорт дерева и **не** ручное выделение в ленте. Только **линейный фрагмент основного русла** между двумя маркерами — результативный отчёт о куске беседы: обсудить с коллегами, оставить на память, переслать, распечатать. Удобный дополнительный инструмент, не ядро продукта.

**Объём:**
- диапазон задаётся парой маркеров «начало» / «конец» (как у сжатия: `list_startable_markers` → `list_reachable_ends` → `resolve_linear_range`, D-063/D-066);
- в документ попадает **линейная цепочка** Q/A между ними; боковые ветки, развилки и «всё дерево» — **вне scope**;
- форматы на выходе: **PDF** и **DOC/DOCX** (Markdown — только внутренняя заготовка, если понадобится, не отдельный продуктовый формат).

**Уже есть в ядре (переиспользовать):**
- `cmd_list_startable_markers`, `cmd_list_reachable_ends`, `cmd_resolve_linear_range`;
- эталон сборки текста диапазона — `compressor/buildTranscript.ts` (заголовок с подписями маркеров, тело В:/О:).

**Плагин:**
- виджет в хосте (как compressor): выбор двух маркеров, превью объёма, кнопка «Сохранить как…»;
- ядро отдаёт нормализованный transcript диапазона; плагин (или узкая `cmd_*`) — генерация файла.

**Вне scope / не делаем:**
- экспорт произвольного выделения мышью;
- экспорт ветки целиком или схемы развилок;
- «архив всей беседы».

Уточняет Q-008: не «дерево наружу», а **линейный отчёт по маркерам**.

### 2. Поиск бесед по тегам — ВЫПОЛНЕНО (см. концепт 3.8)
Реализован поиск по **тегам**: строка поиска в низу `NotebooksPanel`, динамический подбор тегов от 3 символов (`cmd_search_dialog_tags`), плоский список бесед по тегу (`cmd_list_dialogs_by_tag`), переход к беседе без сброса выдачи, сброс только по `×`. Под капотом — справочник тегов (миграция 009) и `tag_id`-связки; теги вводятся по одному с нечёткой подсказкой, плюс LLM-плагин «Тегизатор».

### 3. Секция «Help» в меню приложения
Централизованная справка по **ядру** и **подключённым плагинам**.

Структура:
- пункт меню **Help** (рядом с File / Settings в `MenuBar`);
- вложенное дерево тем: основной функционал (ветвление, маркеры, сжатие, блокноты…) + подразделы по каждому активному плагину;
- переход к хелпу плагина — тот же контракт `HelpDoc`, что уже используют OpenRouter/Gemini (`cap.ui.openHelp`), но каталог собирается хостом из manifest плагинов.

Не путать с текущим оверлеем: сейчас справка открывается только изнутри плагина; нужен **единый вход** и оглавление.

### 4. Плагин «Навигация по дереву»
Визуальный обзор структуры беседы с быстрыми переходами к сообщениям.

**Уровень A (панель, повседневный):** компактное **дерево-список**, **лупа** — поиск по тексту / веткам / маркерам; клик → `ui.focus`. Dock: левая или правая панель.

**Уровень B (окно, позже, D-076):** полноценный **граф** ветвлений — отдельное окно за мостом (D-075), кнопка «Карта дерева» из уровня A.

Отличие от Ctrl+стрелок: карта **всего дерева**, не только линейный обход активной ветки.

### 5. Добавление новой ветки из режима выбора карточек
Если на ответ Ak уже есть, например, три варианта запросов (три карточки), нужна простая возможность задать 4-й вариант прямо из меню выбора карточек — создать ещё одну ветвь, не выходя из режима развилки.
- Использует принцип единого поля ввода: ввод в контексте развилки привязывается к A-узлу-развилке (`forkNodeId`).
- **Сейчас блокер:** в `forkMode` composer скрыт (`{!forkMode && <composer>…}`) — сначала показать поле под карточками.

### 6. Раскладка режима развилки (доработка)
Когда режим выбора карточек **не** вызван:
- сверху — ответ LLM;
- ниже — кнопки дерева и «+»;
- под ними — строка ввода.

Когда режим выбора карточек **вызван**:
- текст ответа LLM уходит вверх на высоту меню карточек;
- ряд альтернативных ветвей — **между** ответом и строкой ввода;
- строка ввода **остаётся видимой** (для п. 5).

Сейчас карточки внизу (п. из «Выполнено»), но composer в развилке прячется — осталось довести до целевой раскладки.

### 7. Retry прямо в заглушке (unanswered_placeholder)
Когда ответ от LLM не пришёл, под холостым Q остаётся серая плашка. Сейчас плашка — некликабельный индикатор: повторная отправка вешает новый Q под заглушку.

Нужно: кнопка «Повторить запрос» на плашке. Не создаёт новый Q, а заново дёргает LLM для существующего холостого Q (`resolve_answer` по тому же `placeholder_id`).

Что тянет за собой:
- `cmd_retry_answer(placeholder_id)` — цепочка до холостого Q, вызов LLM, `resolve_answer`;
- UI: кнопка, loading;
- поведение при повторном провале (плашка остаётся, можно жать снова).

`send_user_message` / `resolve_answer` уже стабилизированы — можно брать в работу.

### 14. Редактирование картинки (image-to-image, мультимодальный ВХОД)
**Сценарий:** «добавь лодку на image-1.png». Nano Banana (`gemini-2.5-flash-image`) умеет image-to-image, так что технически возможно.

**Ключевое: алиас `image-1.png` — чисто наш**, его присваиваем в `persist_llm_media` (`image-{i+1}.{ext}`); реальный файл — `{node_id}-{i}.{ext}`. Модель этого имени **не видела**. Текстом «правь image-1.png» не сработает — надо **физически вернуть БАЙТЫ картинки на вход модели**. Сейчас в LLM уходит только текст (`ChatMessage.content: string`, `buildLlmMessages` картинки не прикладывает).

**Как реально работает:**
1. Пользователь указывает картинку (см. неоднозначность ниже).
2. Программа: алиас → `extra.attachments` → `storage_path` → читает файл → base64.
3. В запрос — мультимодальный вход (картинка + текст):
   - Gemini: `parts: [{ inline_data: { mime_type, data } }, { text }]`;
   - OpenAI/OpenRouter: `content: [{ type:"image_url", image_url:{ url:"data:..." } }, { type:"text" }]`.
4. Модель возвращает **новую** картинку → ловим готовым путём (`extra.attachments` → плашка).

**Подводный камень — алиас не уникален:** `image-1.png` есть в КАЖДОМ ответе с картинками (у каждого A-узла свои). Голая текстовая ссылка неоднозначна → надёжнее **выбор через UI** (клик по плашке → «Редактировать/Использовать как основу»), а не парсинг имени.

**Объём работ:**
- расширить контракт `ChatMessage`/`LlmMessage` под входные картинки (сейчас текст-only);
- сборка мультимодального запроса у обоих провайдеров (inline_data / image_url);
- команда Rust «attachment → base64» (путь уже есть в `extra`);
- UI выбора (кнопка на плашке / чип-ссылка в композере);
- возможности модели: нужен не только image-**выход** (п. 13 `canOutputImages`), но и image-**вход** (`input_modalities` содержит `image`).

**Связь:** следующий логичный шаг после базовой генерации; пересекается с п. 13 (capability) и оркестровкой (инструмент `edit_image` по аналогии с `generate_image`).

Модели image-генерации (напр. `gemini-2.5-flash-image` через OpenRouter) **сами решают**, рисовать или отвечать текстом: мягкие реплики («а можно вариант…», «предложи…») часто дают `images_field=0` (только текст), императивы («Нарисуй…», «Теперь нарисуй…») — картинку. Конвейер перехвата (`message.images` → `extra.attachments` → плашки) работает; пустой результат при таком запросе — решение модели, не баг. При оркестровке учитывать: для роли «генератор изображения» промпт к модели формировать императивно.

### 13. Осознание возможностей модели (image-capability) — НЕ дефект «отлова»
**Контекст/симптом:** на `gemini-2.5-flash-lite` (прямой Gemini API) запрос «нарисуй» дал текст «Вот ваш тропический закат:» и **ни одной картинки**; в консоли — `503` от `generativelanguage.googleapis.com`.

**Разбор (три РАЗНЫЕ вещи, не путать):**
1. **503 — не про картинки.** Временная перегрузка Google на бесплатном тарифе. Авто-ретрай уже есть (`gemini/api.ts`: `RETRY_STATUSES={500,503}`, паузы 800/2000/4000 мс). Браузер логирует первую неудачную попытку, но запрос повторяется — текст в итоге пришёл.
2. **Главная причина пустоты — модель текстовая.** `gemini-2.5-flash-lite` физически не выдаёт изображения, только галлюцинирует «вот картинка». Парсер бессилен: байтов в ответе нет.
3. **Перехват универсален** в пределах наших подключений: OpenRouter → `message.images[]`, Gemini → `parts[].inlineData` (`extractFromGeminiParts`). Для `gemini-2.5-flash-image` он бы сработал, но по прямому Gemini API она требует платного биллинга; Flash-Lite бесплатна, но текстовая.

**Значит, чинить надо не «отлов», а отсутствие у программы знания о возможностях модели.** Данные почти есть:
- OpenRouter: уже храним `outputModalities` (содержит `image`) — см. `openrouter/api.ts`;
- Gemini: в списке моделей есть `supportedGenerationMethods` — по нему/имени вычислить image-способность.

**Что сделать:**
- флаг `canOutputImages` в `ModelFacts` (`llm/types.ts`), прокинуть из обоих провайдеров;
- бейдж «🖼» у image-моделей в выпадашке выбора (в `index.ts` плагинов);
- честная подсказка в ленте: запрос явно про рисунок + активная модель текстовая (или вернула 0 картинок) → ненавязчивая заметка «выбранная модель не генерирует изображения», вместо молчаливого «вот картинка»;
- для оркестровки: роль «генератор изображения» маршрутизируется только на модели с `canOutputImages`.
