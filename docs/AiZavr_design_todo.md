## Архитектурный принцип: единое поле ввода

Ключевая идея, связывающая пункты 2 и 4 ниже. Вместо нескольких полей ввода (обычный промпт + всплывающее поле ветвления + потенциальное поле для новой карточки) — **одно поле ввода внизу экрана**. Меняется только «точка привязки» результата в зависимости от контекста:

- обычный промпт → привязка к листу активной ветки;
- ветка по кнопке «+» → привязка к выбранному A-узлу;
- новая карточка в режиме развилки → привязка к тому же A-узлу-развилке.

Поле одно, логика привязки переключается контекстом. Это упрощает и UI (нет дублирующих полей), и код. Стоит держать как принцип при рефакторинге ввода.

## Список замечаний

### 1. Экспорт беседы (функционал, не только дизайн)
Возможность выделить часть беседы и кнопкой выгрузить её файлом. Форматы: PDF, DOCX и др.
- Пересекается с Q-008 в spec (экспорт ветки/дерева наружу).
- Отложено: это полноценная фича, не правка дизайна. Зафиксировать здесь как UI-вход в неё (выделение + кнопка), реализация — по роадмапу (v0.4, экспорт в Markdown как первый шаг).

### 2. Кнопка «+» использует основное поле ввода
Сейчас при нажатии «+» под A-узлом выскакивает отдельное поле ввода. Заменить на:
- при нажатии «+» под ответом An низ этого ответа автоматически подскролливается так, чтобы оказаться прямо над строкой ввода;
- далее используется то же основное поле ввода, что и для обычных запросов;
- второе (всплывающее) поле убрать.

### 3. Карточки вариантов — внизу экрана
Ряд карточек альтернативных запросов выводить внизу, под окончанием ответа LLM, а не вверху (как сейчас, между индикаторами глубины и сообщениями).

### 4. Добавление новой ветки из режима выбора карточек
Если на ответ Ak уже есть, например, три варианта запросов (три карточки), нужна простая возможность задать 4-й вариант прямо из меню выбора карточек — создать ещё одну ветвь, не выходя из режима развилки.
- Использует тот же принцип единого поля ввода (см. выше): ввод нового запроса в контексте развилки привязывается к A-узлу-развилке.

### 5. Раскладка режима развилки
Когда режим выбора карточек НЕ вызван:
- сверху — ответ LLM;
- ниже — кнопки дерева и «+»;
- под ними — строка ввода.

Когда режим выбора карточек вызван:
- текст ответа LLM уходит вверх на высоту меню карточек;
- ряд альтернативных ветвей располагается между ответом и строкой ввода.

То есть карточки вклиниваются между ответом и полем ввода, а не перекрывают контент и не висят сверху.

### 6. Мягкое удаление ветви (архитектура, не дизайн)
Удаление ветви делается «фиктивным»: узлу ставится признак `is_deleted`, ветвь скрывается из выдачи, физически ничего не удаляется (в духе D-004 — ничего не теряется, потом можно сделать «корзину»/восстановление). Вход в UI — пункт «Удалить» в контекстном меню карточки (символ ⋮); каркас меню уже готов, пункт добавляется после реализации.

Что тянет за собой (нужно проработать перед реализацией):
- поле `is_deleted` в `nodes` → миграция 004;
- фильтрация удалённых в `get_children`, `get_branch`, в подсчёте для иконки развилки (`children_count` — считать ли удалённых);
- что делать с `active_child`, если удалена активная ветвь — переключить на живую сестру; какую;
- случай удаления единственной оставшейся ветви.

Кандидат в Q-0xx в spec при следующем обновлении — это открытый архитектурный вопрос, а не правка дизайна.
