## Что сделано (фронтенд)

### 1. Единое нижнее поле ввода — composer (реализация D-087)
- Однострочный `<input>` заменён на `<textarea>` с переносом слов (`white-space: pre-wrap; overflow-wrap: break-word`).
- **Перетаскиваемая верхняя граница**: тонкая полоса-хваталка сверху, тянем вверх/вниз — высота 60px..60% окна. Состояние высоты — локальный `composerHeight`.
- **Enter** — отправить, **Shift+Enter** — перенос строки.
- **Убрано встроенное поле альтернативного запроса** под A-узлами (`branch-input-row` и стейт `branchInput` удалены целиком).
- Альтернативный запрос теперь вводится в ТОМ ЖЕ нижнем поле: при входе в ветвление (кнопка «+») в поле появляется **префикс «Альтернативный запрос:»** (синяя подсветка рамки), курсор справа от него. Отмена — «✕» / Escape. Кнопка «+» активного узла подсвечена.

### 2. Автоскролл ленты + кнопка «вниз»
- «Прилипание» к низу (`stickToBottom`): пока пользователь не отлистал вверх (>40px от низа), лента держит последнюю строку отправленного над полем ввода. Срабатывает на любое изменение (ответ LLM, запрос, заглушка «Ответ не получен», «Думаю…»). `useLayoutEffect` — без мигания; в deps учтена и `composerHeight` (поле растёт — низ подтягиваем).
- **Круглая стрелка ↓** справа внизу ленты: видна только когда отлистали вверх; клик — доезжаем до низа и снова приклеиваемся. Лента обёрнута в `.messages-wrap` (relative) ради позиционирования кнопки.

### 3. Позиционирование при ветвлении
- При нажатии «+» лента подводит **последнюю строку базового запроса этой ветки** (вопрос сразу за выбранным A-узлом; если его нет — сам A-узел) к низу области сообщений (`scrollIntoView block:"end"`), автоскролл отпускается, фокус уходит в поле с префиксом. Старый вопрос виден прямо над «Альтернативный запрос:».

### 4. Переработка навигации Ctrl+↑ / Ctrl+↓ (заметно расширяет логику сессии 7)
Единая логика **«точек остановки»**: узел является точкой, если это **маркер** ИЛИ **развилка** (`children_count − deleted > 1`). Узел встаёт нижней строкой к низу окна (`collectNavStops` считает целевой `scrollTop` в координатах контента с поправкой на текущий скролл; `scrollNodeBottomToView` пересчитывает после смены раскладки).
- **Ctrl+↑** — ближайшая точка ВЫШЕ текущего положения; маркер → просто стоп (в нижней строке), развилка → раскрывает плашки альтернатив **внизу** окна (раньше были вверху). Повторное нажатие едет дальше вверх — больше **не застревает** на ближайшей (это и просил идеолог в сессии 7, долг #навигация закрыт).
- **Ctrl+↓** — то же вниз; если точек ниже нет — к самому низу и снова прилипает.
- **Плашки развилки переехали ВНИЗ**, на место поля ввода (composer прячется в режиме развилки). Управление:
  - **←/→** — выбор варианта; **Ctrl+←/→** — к крайнему левому/правому;
  - **Enter** или **↓** (без Ctrl) — провалиться в выбранную ветку;
  - **Ctrl+↑/↓** в режиме развилки — НЕ проваливаемся, едем дальше к следующей точке (плашки закрываются);
  - **F2** — переименовать.
- **Провал в ветку — только на ПЕРВОЕ сообщение** ветки (`block:"start"`), не в самый низ; дальше вниз — кнопками/стрелкой. (Раньше Ctrl+Down в развилке = выбор; теперь выбор — Enter/↓, а Ctrl+↑↓ — навигация.)

### 5. Убраны индикаторы глубины
Голубые вертикальные полоски (`depth-indicators`/`depth-bar`) удалены целиком — разметка, стейт `depth`, интерфейс `DepthIndicators`, **вызов `cmd_get_depth_indicators`** с фронта и стили (вкл. dark). Бэкенд-команда осталась — фронт её просто не дёргает (неинформативна, решено в этой сессии).

## ГЛАВНОЕ ДЛЯ СЛЕДУЮЩЕЙ СЕССИИ — организация блокнотов через плагины

Обсуждено и зафиксировано как **оценка реализуемости** (кода не писали). Идеолог НЕ хочет вшивать блокноты «намертво» в ядро: организация, переподчинение, анализ, «смарт»-блокноты, облако/синк, шифрование паролем, публичность — всё это должно быть внешним модулем.

### Вердикт
Реализуемо и архитектурно «родно» — лучше, чем вшивать в ядро. НО «через плагины» ≠ «ноль ядра»: нужен **тонкий стабильный якорь в ядре**, а вся начинка — снаружи. Развилка не «можно ли», а «где граница».

### Почему ложится на текущую архитектуру
Примитивы уже есть: узкий контракт `types.ts`; курируемые капабилити вместо generic-доступа (нет `invoke`/`fetch`/ручки к БД — каждая возможность тонкая обёртка над `cmd_*`, D-072); прецедент «непрозрачный blob плагина, ядро хранит/релеит» (`extra.compression` + provenance, D-065) — ровно под смарт-правила; секреты через keychain ядра (`secrets`); два класса плагинов уже сосуществуют (виджеты + `LlmProvider`-реестр) — добавить третий класс это паттерн; курс на изоляцию за мостом (D-075) — место для выдачи тяжёлых прав (сеть/крипто).

### Что УЖЕ есть в схеме БД (migration 001) — фундамент наполовину готов
- `notebooks (id, parent_notebook_id → иерархия, name, knowledge_card_id, system_prompt, default_model_overrides, created_at)`.
- `dialogs.notebook_id → notebooks(id) ON DELETE SET NULL` (+ индекс `idx_dialogs_notebook`).
- `knowledge_cards (id, notebook_id, sections JSON, …)`.
- Индекс `idx_notebooks_parent`.
- **Нет CRUD-команд блокнотов** — только `dialogs` несут `notebook_id`. То есть таблицы есть, поведения нет.

### Принцип: механизм — в ядре, политика — в плагине
- **Ядро (механизм, тонко):** канонная сущность «блокнот» — id/имя/родитель/принадлежность диалога, инварианты (нет циклов иерархии, целостность FK), события об изменениях, read-проекция для панели и плагинов (по образцу `WidgetFacts`).
- **Плагины (политика):** организация/переподчинение как UX, смарт-правила, аналитика, облако/синк, шифрование, шаринг.

Почему якорь обязателен: `dialogs` уже ссылается на `notebooks` по FK, левая панель «Блокноты» — хром ядра, нужен единый источник правды «в каком блокноте диалог». 100% в плагине → удаление плагина осиротит диалоги и сломает панель. Тонкий якорь ≠ «вшить всё намертво» (намертво боимся тяжёлой начинки — её и выносим).

### Пофичёвая оценка (реализуемость / что нужно от ядра / сложность)
- **Формировать/переподчинять/организовать** — Да / CRUD + reparent + назначение диалога + события; инварианты иерархии / **низкая**.
- **Анализировать** — Да / несколько read-капабилити (диалоги в блокноте, статистика); можно кормить LLM через `model.call` (роли, v0.2) / **низкая**.
- **«Смарт»-блокноты** — Да / `kind: static|smart` + непрозрачное правило-blob (как provenance), ядро хранит и материализует/обновляет членство, плагин вычисляет / **средняя**.
- **Облако: сохранение/синк** — Да, самый тяжёлый кусок / **канонный формат снапшота** (экспорт/импорт блокнота+диалогов+дерева) + модель разрешений на сеть + медиатор секретов; вероятно новый класс «provider»-плагина / **высокая**.
- **Шифрование паролем** — Да, в связке с ядром / понятие «заперто» в проекции: ядро хранит шифртекст и не отдаёт плейнтекст до разблокировки (плагин даёт алгоритм/KDF + UX). Прецедент — приватные диапазоны D-078 (редакция при проекции) / **средняя, security-sensitive**.
- **Публичность/шаринг** — Да / тот же снапшот + редакция приватного/зашифрованного + provenance + сетевой бэкенд / **высокая (зависит от синка)**.

### Узкие места (ядровые РЕШЕНИЯ, даже если реализации — плагины)
1. **Канонный формат снапшота** блокнота как единица для синка/шаринга/экспорта. Задать рано, иначе каждый плагин изобретёт свой и они не скомпонуются.
2. **Модель разрешений** сеть/секреты (allowlist доменов, медиатор HTTP или нетто-капабилити за мостом). Сейчас generic-сети у плагина нет намеренно.
3. **Правило locked/редакции-проекции**: ядро обязано уметь «не эмитить» закрытый контент. D-078 это уже предвосхищает.

### Фазовый план
- **Фаза 1 (дёшево, можно начинать):** ядровый CRUD/иерархия/членство блокнотов + события + read-проекция; плагин «Менеджер блокнотов» для организации/переподчинения, смарт через непрозрачное правило, аналитика через чтения. Закрывает «организовать/анализировать/смарт» почти без лок-ина.
- **Фаза 2 (позже, тяжелее):** формат снапшота + класс «provider»-плагина + модель разрешений и locked-проекция → разблокирует синк/шаринг/шифрование без переписывания Фазы 1.

### Конкретные первые шаги по коду (Фаза 1, с рустового конца)
1. **`notebooks/mod.rs`** (vertical slice, владеет таблицей): `create_notebook`, `rename_notebook`, `move_notebook` (reparent, с проверкой отсутствия цикла), `delete_notebook` (политика для диалогов внутри — на SET NULL уже есть FK), `list_notebooks` (дерево/плоско), `assign_dialog_to_notebook`, `list_dialogs_in_notebook`.
2. **`lib.rs`**: соответствующие `cmd_*`.
3. **Капабилити блокнотов** в `host/capabilities.ts` + контракт в `types.ts` (тонкие обёртки над `cmd_*`, НЕ generic): `notebooks.create/rename/move/assign/list`. + новый `CapabilityName 'notebooks.*'`.
4. **Read-проекция/факт** для панели: список блокнотов реактивно (по образцу `WidgetFacts`/событий дерева).
5. **NotebooksPanel** наполнить: дерево блокнотов, перетаскивание/переподчинение, привязка текущего диалога. Сейчас там заглушка — точка втыкания готова.
6. Смарт-правило: `kind` + opaque-blob; в Фазе 1 плагин вычисляет членство по чтениям ядра.

### Главный риск
Не реализуемость, а **расползание границы**: пока плагин в основном webview, соблазн дотянуться до `fetch`/generic-`invoke` сломает изоляцию (и развалится на D-075). Дисциплина: каждая новая сила — именованная медиируемая капабилити, не дырка.
