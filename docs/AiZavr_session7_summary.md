## Что сделано: устойчивый поток отправки

### Проблема
Старый `doSend` создавал Q-узел и A-узел двумя раздельными `create_node` с
вызовом LLM между ними. В окне между Q и A приложение могло упасть / связь
оборваться / пользователь уйти в другую ветку → холостой Q оставался без
ответа. Это ломало регулярность дерева (Q→A) для сжатия, навигации и любого
потребителя линейного пути. «Лечить перед сжатием» — костыль, который пришлось
бы повторять перед каждым потребителем.

### Решение
**Заглушка-ответ ставится ВСЕГДА, атомарно с Q, ДО вызова LLM.**

Жизненный цикл:
```
юзер шлёт Q
  → cmd_send_user_message: создаёт Q + unanswered_placeholder атомарно,
    лист диалога = заглушка
  → фронт зовёт LLM
     ├─ ответ пришёл  → cmd_resolve_answer: UPDATE заглушки
     │                  (node_type → assistant_message, content, model)
     └─ ответа нет (краш/обрыв/уход) → заглушка остаётся, структура регулярна
```

- `unanswered_placeholder` — служебный `node_type` (по структуре, не по тексту;
  тот же принцип, что `compression_placeholder` D-061). content пустой.
- `resolve_answer` проверяет, что узел ВСЁ ЕЩЁ заглушка (защита от двойной
  доставки/гонок) — иначе ошибка, узел не трогается.
- Заглушка существует на КАЖДОЙ отправке; при успехе живёт миллисекунды
  (создали → перезаписали). Это норма, не баг — видно в БД при отладке.
- В UI холостая заглушка (ответ не пришёл) = серая некликабельная плашка
  «Ответ не получен». При нормальном потоке плашка не видна (крутится loading,
  заглушка перезаписывается до перерисовки).

### Сборка LLM-цепочки (buildLlmMessages, фронт)
Правило: встретив `unanswered_placeholder` В СЕРЕДИНЕ ветки — срезать и его, и
Q над ним (холостой Q из прошлого сбоя; для модели этой пары не было). Заглушку
НА КОНЦЕ ветки (текущий ожидающий запрос) НЕ срезать — её Q обязан уйти в LLM.
[Этот баг — срез текущего Q — был словлен и починен в этой сессии.]

Примечание: правило живёт на фронте. Сработает только при перезагрузке с
холостым Q в дереве. Перенос в ядро (`get_branch_for_llm`) отложен — переедет
вместе со сборкой цепочки, когда будем делать сжатие (там добавятся S-узлы).

### Файлы
- `db/mod.rs`: + `SendResult`, `send_user_message`, `resolve_answer`.
- `lib.rs`: + `cmd_send_user_message`, `cmd_resolve_answer`.
- `App.tsx`: `doSend` и `sendBranch` переведены на новый поток; `loadBranch`
  показывает заглушки плашкой; `buildLlmMessages` срезает холостые пары.

## Что сделано: маркеры (D-058)

### Бэкенд (из этой сессии, скомпилирован)
- **Миграция 005**: таблица `markers (id, node_id, label, comment, created_at)`
  + индекс `idx_markers_node`.
- **`markers/mod.rs`** (новый модуль, vertical slice — владеет таблицей сам):
  - CRUD: `create_marker` (только на assistant_message — инвариант D-058 держит
    ядро, БД выразить не может), `get_marker`, `get_markers_for_node`,
    `list_markers_for_dialog`, `update_marker`, `delete_marker`.
  - Целостность D-067: `is_node_referenced_by_compression` — удаление маркера
    запрещено, пока на его node_id ссылается compressed_summary
    (extra.compression.start/end). Признак ВЫЧИСЛЯЕТСЯ, не хранится. В MVP
    всегда false (S-узлов нет), но хук правильный.
  - Топология D-066: `list_startable_markers` (маркер на видимом не-листе),
    `list_reachable_ends` (видимые потомки = маркеры или листья; тип
    `ReachableEnd`), `resolve_linear_range` (инклюзивный линейный путь,
    start первым, ошибка если не на одной линии или задевает удалённый узел).
- **`lib.rs`**: 9 marker-команд (CRUD + топология + проверка блокировки).

### UI маркеров (этой сессии)
- Кнопка-чекбокс `⚑` под A-узлом (рядом с ⑂/+). Тогл: нет маркера → ставит,
  есть → снимает.
- Нумерация `#N`: max существующих по активной ветке + 1. Дырки от удаления НЕ
  переиспользуются (монотонный рост). Имя не редактируется (чтобы не вводить
  контроль уникальности).
  - ОГРАНИЧЕНИЕ MVP: номер считается по загруженной активной ветке, не по
    всему диалогу. Маркеры в боковых ветках могут дать дубль #N в разных
    ветках. Приемлемо для одной рабочей ветки; строгая монотонность по диалогу
    потребует cmd_list_markers_for_dialog вместо обхода messages.
- Комментарий: поле при постановке (необязательное). Серый обрезанный текст
  справа от `⚑ #N`. Двойной клик → textarea (динамическая высота),
  Enter — сохранить и схлопнуть, Shift+Enter — перенос, Esc — отмена.
- Снятие маркера удаляет и комментарий (никакого «следа» — решено в этой
  сессии; «погашенный маркер с серым следом» отложен в TODO #9).

### Сценарий-мотивация (со слов идеолога)
Маркер как навигационный якорь: «получил ответ, от которого пойду вбок —
ставлю маркер, гоню диалог дальше на 20 шагов, потом одним прыжком возвращаюсь
к маркеру и ответвляюсь». Навигация Ctrl+Up к маркеру — СЛЕДУЮЩАЯ сессия
(см. ниже), не сделана.

## Починенные баги OpenRouter (`llm/openrouter.rs`)
1. **Обязательный `usage`** валил парсинг всего ответа («error decoding response
   body»). Сделаны `Option` + `#[serde(default)]` для usage/model/content.
2. **Слепая ошибка**: тело ответа теперь читается как текст, проверяется
   HTTP-статус — при ошибке возвращается реальный JSON OpenRouter, а не
   «error decoding response body». Добавлены заголовки HTTP-Referer / X-Title.

## Следующие шаги (код)

### Навигация по маркерам — Ctrl+Up (НЕ начата, обсуждена)
Идеолог хочет: Ctrl+Up прыгает к ближайшей развилке ИЛИ ближайшему маркеру
вверх по активной ветке; в случае маркера — подгоняет «хвост» маркированного
ответа к низу экрана, чтобы удобно нажать + и ответвиться.
ВАЖНО: идеолог хочет ЗАОДНО переделать сам `handleCtrlUp` — чтобы повторное
нажатие «схлопывало» текущий уровень и поднимало к развилке следующего уровня
(сейчас застревает на ближайшей). Это переписывание всего handleCtrlUp
(уровни + маркеры + перебор) — делать ОДНИМ заходом, не вмешивая маркеры в
старый код по частям.

### Сжатие — `compression/mod.rs` (НЕ начато)
Маркеры и топология готовы — фундамент под сжатие заложен. Осталось:
- `trait Compressor { async fn compress(path: Vec<DbNode>) -> CompressionResult }`
  где `CompressionResult { summary_text, placeholder_text: Option<String>,
  plugin_id, plugin_version, algorithm, params }`.
- `attach_compressed(...)` — создаёт S (compressed_summary, Q-слот, сестра к
  верхнему A) + заглушку (compression_placeholder, A-слот) + пишет
  extra.compression (D-065).
- Референсный Compressor-заглушка. Алгоритм дайджеста СОГЛАСОВАН в сессии 6/7:
  - путь регулярен `A(якорь) → Q → A → Q → A(конец)`, всегда нечётной длины;
  - `path[0]` (якорь) — строка `[предыстория] A: <первые 10 слов>`;
  - далее по парам: `Q: <первые 5 слов>` / `A: <первые 10 слов>`, каждое с
    новой строки; слова, не символы; «…» если обрезано;
  - заглушка (path[0]) НЕ дублируется в summary как контент — она остаётся
    дословно выше по дереву (D-063); в дайджест идёт только как помеченная
    префиксом предыстория;
  - на входе компрессор ВАЛИДИРУЕТ регулярность пути (нечётная длина, чередование,
    первый/последний = A) — внятная ошибка вместо паники на индексах;
  - placeholder_text = None (заглушка в модель не идёт);
  - algorithm = "noop-digest", plugin_id = "reference-noop", version "0.1.0".
- Команды lib.rs: cmd_compress_range и пр.
- UI: двухшаговый выбор пары маркеров (D-068), кнопки L1/L2/L3, preview (D-008).

## Замечание по процессу (для следующих сессий)
Крупные правки в JSX (App.tsx) лучше отдавать ЦЕЛЫМ ФАЙЛОМ на замену, а не
фрагментом: при ручной вставке кусков отступы перебиваются (табы/пробелы
вперемешку), теряется скобка, ловится «Unexpected token». Мелкие правки
(функция, пара строк) — фрагментом с адресом (файл, строки, действие).
